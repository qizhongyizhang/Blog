数据库

写在前面：本文主要记录数据库基础知识

数据库系统

数据库系统(DBS：Database System)：数据库系统 = 数据库管理系统 + 数据库 + 管理员；数据库系统包括数据、硬件、软件(基础软件：操作系统，数据库管理系统)和人员(用户、管理员)
数据库管理系统(DBMS：Database Management System)：数据库管理软件

  • 基于共享文件系统的DBMS：通常用于桌面用途，不用于高端或更关键的应用，如Microsoft Access、FileMaker等
  • 基于客户机—服务器的DBMS：服务器部分负责所有数据访问和处理，客户机将服务器处理好的数据返回给用户，服务器与数据打交道，客户机与客户打交道，如MySQL、Oracle以及Microsoft SQL Server等

数据库：按照数据结构来组织管理数据的仓库，对大量数据进行高效的管理，是数据库管理系统所管理和操作的数据的容器

常见的数据库

关系型数据库：SQL Sever，MySQL，Oracle，PostgreSQL，DB2，SYBase等
非关系型数据库：MongoDB，Redis等

管理员(DBA：Database Administrator)：操作和管理数据及数据库管理软件的人

发展

  • 1960s：文件系统
    • 缺点：不一致、费时、完整性、安全问题
  • 数据库系统
    • 层次型数据库(IBM)：最古老的数据库类型，数据被组织成具有强制父子关系的树形结构，父与子是一对多关系
    • 网络型数据库
    • 关系型数据库：是建立在关系模型(二维表)上的数据库系统，要求数据记录具有相同的关系结构，基于几何理论和关系代数，常见的关系型数据库有Oracle、SQL Sever、DB2、MySQL、PostgreSQL等
    • 面向对象
    • 非关系数据库：是建立在非关系模型上的数据库系统，有列式数据库、空间数据库、键值对数据库等
  • 知识型数据库

关系型数据库

关系型数据库（RDBS， Relational DataBase System）是建立在关系模型(二维表)上的数据库系统，要求数据记录具有相同的关系结构，基于几何理论和关系代数

  • 数据结构可以规定：同类数据，结构一致，二维表；
  • 数据间的关系可以设置：日常生活中的实体和实体的属性保存到数据中

非关系型数据库

非关系型数据库（NoSQL: Not Only SQL）是建立在非关系模型上的数据库系统
列式数据库：某列数据仅有唯一值，压缩冗余数据，常用于商务智能
文档数据库

MongoDB
ElasticSearch

键值对数据库/Key-Value数据库

Redis (Remote Dictionary Server)，高速缓存服务，单线程+异步IO (多路I/O复用：Tornado、Node.js)，常用于：高速缓存服务（用户经常访问的数据(热数据)从数据库搬到内存），实施排行榜，投票点赞，消息队列等

图数据库

模式

  • 外模式：子模式/用户模式，用户层，用户使用的局部数据的逻辑结构和特征
  • 模式：概念模式/逻辑模式，概念层，存储全体数据的逻辑结构和特征
  • 内模式：存储模式，物理层，数据的物理结构和存储方式的描述

SQL

SQL：Structured Quary Language，结构化查询语言，专门用来与数据库通信的语言；SQL语句以分号（；）分隔，SQL语句不区分大小写，推荐对所有SQL关键字使用大写对所有列和表名使用小写

  • 发展

    1. 1972：System R项目中提出Sequel
    2. 结构化的查询语言：标准86、92、99、03

  • 分类

数据定义(DDL：Data Definition Language数据定义语言)：进行建库建表等操作
数据操作(DML：Data Manipulation Language,数据操作语言)
数据查询(DQL：Data Query Language,数据查询语言)
数据库管理(DCL：DataBase Control Language,数据库控制语言)

  • 优点

SQL不是某个特定数据库供应商专有的语言，所有重要的 DBMS 都支持 SQL
SQL简单易学，语句全都是由描述性很强的英语单词组成，且数量较少
是一种强有力的语言，灵活使用其语言元素，可以进行非常复杂和高级的数据库操作

关系型数据库

关系型数据库：RDBS: Relational DataBase System

基本概念

  • 数据库(DataBase)：存储数据的容器或集合
  • 模式(Schema)：是数据库的数据库对象的一个子集，用来将包含敏感数据的对象和普通用户分离
  • 表(Table)：用于存放数据，是某种特定类型数据的结构化清单，数据表由数据行和数据列构成
  • 列(Column)/字段(Field)： 表中的一个字段，多个列水平拼接成一个数据表，可以规定数据的存储类型(规范数据的存储)
  • 数据类型(Datatype)：所容许的数据的类型，限制（或容许）该列中存储的数据
  • 行(Row)：用于记录数据，多行竖直拼接成一个数据表，一行数据也称一条记录(Record)
  • 键(Key)：唯一标识实体实例的一个或多个属性的集合
    • 单一键：唯一识别实体实例的是一个属性
    • 组合键：唯一识别实体实例的是一组属性
  • 主键(Primary Key)：数据的“身份证”，用于唯一确定一条记录
    • 任意两行都不具有相同的主键值；
    • 每个行都必须具有一个主键值(主键列不允许为空)
  • 外键(Foreign key)：一个实体的属性依赖于另一个实体的主键
  • 域(Domain)：属性的可能取值范围，属性不应该含有域以外的值
  • SQL(Structured Quary Language：结构化查询语言)：用来管理数据的语言
  • 关系(Relational)：关系模型
  • 数据库抽象：用来调用数据库函数的通用应用接口，如ODBCC、JDBC等

关系模型

关系模型：Entity-Relatonship 模型设计

实体

实体是区别于其他事物的一个事物，一个对象，由属性取值唯一确定，用表存储：属性取值具有原子性，元组无序

  • 单一属性
  • 组合属性：姓/名
  • 单值属性
  • 多值属性：电话号码
  • 派生属性：通过属性间的运算可获得

关系

涉及实体数量
涉及实体数量称为元数

一元关系：也称递归关系，或自我引用关系，一对多描述层级关系，如课程需要先导课，虚线表示；多对多描述网络或图表，实线表示
二元关系：最常见
三元关系：涉及三个实体，基于事实建模，学生可以再特定时期选择一门特定课程

实体间的关联关系

一对一：一对一/一对一，两个实体表存在相同的主键字段，垂直分割（常用/不常用）；
一对多：一对多/一对一，一个实体对应多个其他实体，在n端实体增加字段，指向对应实体
多对多：一对多/一对多，多个实体对应多个实体，利用中间表保存实体间的对应关系，映射成两个一对多关系，关系表指向实体表

关系代数

  • 选择：select
  • 投影：project：消除重复行
  • 并：union：列数相同，域相同，消除重复行
  • 差：set difference
  • 笛卡尔积(cartesian product)，又叫直积，两集合内数据所有可能的排列组合。两个二维表的没有条件的内连接（交叉连结：cross join）就是笛卡尔积。
  • 重命名：rename
  • 交：保留两表相同的部分
  • 自然连接：连接具有相同属性的两表，属性是两表属性不重复的集合，保留取值相同的元组
  • 外连接
    • 左外连接：共有属性列以左表为准拼接两表，允许右表属性缺失
    • 右外连接：共有属性列以右表为准拼接两表，允许左表属性缺失
    • 全连接：共有属性列由两表集合为准，允许缺失
  • 除：连接具有相同属性的两表，属性是两表的差，保留元组的对应属性的取值完全包含另外一张表

设计原则

  • 属性不能有关系
  • 一个实体不能作为另一个实体的属性：弱实体
  • 联系可以有标识符
  • 实体联系可以作为实体
  • 先构建实体/关系，后添加属性

数据库构建原则

PRISM原则

  1. 性能和易用性（Performance of Ease of Use）：确保用户可以快速、轻松的访问数据，提高应用程序和数据的业务价值
  2. 可重用性（Reusability）：确保数据库结构在适当情况下可以被多个应用重复使用
  3. 完整性（Integrity）：数据完整的表述业务含义和、价值和有效状态，且能监测并报告数据的完整性约束违规事件
  4. 安全性（Security）：始终及时向授权用户提供真实准确的数据，且满足利益相关方的隐私要求
  5. 可维护性（Maintainability）：确保创建、存储、维护、使用数据的成本不超过其对组织的价值

键/码

超码可以唯一确定一个元组，最小的超码为候选码；主码从候选码中挑选；外码，连接到另一个表的的主码

column_name data_type primary key auto_increment;
// 自增长：属性为整型，属性有索引，可以设置初始值
primary key (column_name)

完整性约束

  • 实体完整性：每个实体都是独一无二的，没有冗余，且主码不为空
  • 参照完整性：外码必须匹配主码，外键约束
  • 域完整性：也叫数据完整性，存储的数据都是有效的，数据列的取值在域内（数据类型、数据长度、默认值约束、非空约束、检查约束）
  • 用户定义完整性：用户自定义的完整性约束，不同的关系数据库系统根据应用环境不同，需要一些特殊的约束条件，反映某一具体应用必须满足的语义需求

列属性约束

列属性约束：保证表的结构和数据的正确性和稳定性

  • 唯一性和主键约束(UNIQUE,Primary Key)：表中的一个或多个字段，用于惟一地标识表中的某一条记录，要求主键的值不能重复。
  • 外键约束(Foreign Key)
  • 检查约束(CHECK)
  • 空值约束(NOT NULL)：表示没有值，什么都没有。与任何数据不同。如果一个列不允许为空，但是在赋值时，没有为该字段赋值，则会报错。使用 is null or is not null来判断，不能直接比较=/>null
  • 默认值约束(DEFAULT)：为列指定一个默认，当该字段不存在值时，会被启用。默认值必须为一个常数，不能为一个函数或一个表达式。如果该列可以为空，则默认值为null。有些列是不能有默认值的：Blob，text。时间戳类型，可以设置一个特殊的默认值 CURRENT_TIMESTAMP。在当列不存在或者传递的值为null时，使用当前的时间戳。

逻辑设计

  • 任务：E-R图转换成关系表
  • 组合属性由组合值代替
  • 派生属性省略
  • 多值属性另外生成一张关系表
  • 保留单值属性
  • 一对多/多对一：对多方的属性加入一方的主码
  • 多对多关系：保留实体的主码和自身属性
  • 弱实体转换为关系要增加依附实体的主码

函数依赖：A->B

阿姆斯特朗公理

  • 包含规则：Y属于X，则X可以推出Y（X、Y是R的属性组）
  • 传递规则：X决定Y且Y决定Z，则X决定Z
  • 增广规则：X决定Y，则XZ决定YZ
  • 推1合并规则：X决定Y，X决定Z，则X决定YZ
  • 推2分解规则：X决定YZ，则X决定Y，X决定Z

闭包

  • 函数依赖集闭包：由函数依赖集F所能推出的所有函数依赖的集合称为F的闭包
  • 属性集闭包：属性集a在函数依赖集F下所能推出的所有属性的集合称为a的闭包

正则覆盖

  • 覆盖：函数依赖集F包含函数依赖集G，则F覆盖G
  • 等价：F可覆盖G，G可覆盖F，则F、G等价
  • 最小函数依赖集
    • 分解函数依赖集右侧，化解为单属性
    • 去除函数依赖集左侧的冗余属性：尝试去除属性集的某一属性
    • 去掉多余的函数依赖（可由其他函数依赖推到而出）
    • 函数依赖集C与F等价，没有函数依赖有多余的属性，函数依赖左部不重复，则C是F的正则覆盖

范式

范式（NF，Normal Form），数据库设计范式，表示设计关系型数据库需要满足某种级别的一些规则和要求。分为1NF，2NF，3NF，BCNF，4NF，5NF，6NF，要求是从低到高逐渐递增。关系型数据库必须满足1NF，通常达到3NF就可以满足要求

  • 1NF：要求数据库表的每一列都是不可分割的基本数据项，同一列中不能有多个值，即实体中的某个属性不能有多个值或者不能有重复的属性
  • 2NF：在满足1NF的基础上，要求表中的每条记录必须被唯一的区分，实体属性应该完全依赖于主关键字。而不能是对主关键字形成部分函数依赖。常采用单关键字主键满足要求，即为每一条记录一个字段用于表示其主键
  • 3NF：在满足第二范式的基础上，要求不能出现传递依赖，任何字段不能由其他字段派生出来，即不能出现属性依赖于非主属性。通过将传递依赖的数据单独建立二维表，保存数据解决
  • BCNF：解决较差的复合侯选建
  • 4NF：将三元关系都分解成二元关系
  • 5NF：将实体内部依赖关系分解成二元关系

    优缺点

    可以尽量减少冗余，使得更新快，体积少
    对于查询需要进行多表关联，减少写效率，增加读效率，更难进行索引优化

    采用反范式化（降低表的范式），可以减少表关联，更好进行索引优化；但会存储较多的冗余数据，数据修改成本较高

事务

事务是指逻辑上的一组操作，组成这组操作的各个单元，要不全成功要不全失败

ACID特性

  • A（atomicity）原子性：事务是一个不可分割的工作单位，要么不做，要么全做
  • C（consistency）一致性：事务执行结果正确，事务前后数据的状态必须保持一致
  • I（isolation）隔离性：多个用户并发访问数据库时，一个用户的事务不能被其它用户的事物所干扰，多个并发事务之间的数据要相互隔离
  • D（durability）持久性：事务完成后，结果直到下一次修改前，永久保存
    redo log 在数据库崩溃时恢复数据，保证数据的一致性和持久性
    undo log 用于对事务回滚时进行撤销，也用于隔离性的多版本控制

实现

  • 要求是事务支持的表类型
  • 执行一组相关的操作前开启事务(Start transaction)
  • 整组操作完成后，都成功，则提交(Commit)，如果存在失败，选择回滚(Rollback)，则会回到事务开始的备份点
    • 自动提交:Autocommit=0|1

状态

  • 活动状态：active，开始执行
  • 部分提交状态：partially committed，指令执行结束
  • 失败状态：failed，不能正常执行
  • 终止状态：aborted，失败回滚，恢复成初始状态
  • 提交状态：committed，结果提交给数据库之后
  • 终结状态：Terminated，终止态或提交态

隔离级别

RU：读未提交，产生脏读
RC：读已提交，产生不可重复读
RR：可重复读，产生脏读
Serializable：不存在事务并发问题，为读写都加锁，效率不高

并发控制

并发冲突类型

  • 写写冲突：丢失修改，同时更新
  • 读写冲突：不可重复读(A会话两次查询，查询的结果记录数一致，记录内容不一致)，B会话在两次查询之间发生 update 语句
  • 写读冲突：读脏数据(A会话读到B会话未提交的数据)，回滚
    幻读：A会话两次查询，查询的结果记录数不一致，B会话在两次查询之间发生 insert 或 delete 语句

锁机制

  • 排它锁：其他事务不可进行读写操作
  • 共享锁：所有事物可读，不可写
  • 死锁：互相等待；饿死：持续等待
  • 两阶段封锁协议：增长阶段：只允许增加锁，不允许释放；消减阶段：只允许释放，不允许增加锁

字符集

字符集是一套字符与字符编码的集合，常用字符集见 链接文档  的字符编码部分

Show character set; 可以获得支持的所有字符集
character set gbk collate gbk_chinese_ci;设置字符集，也可在配置文件中修改
 - character_set_database：默认数据库字符集（数据）
 - character_set_sever：默认mysql服务器字符集
 - character_set_system：标识符的字符集（库名，表名）

校对规则：当前字符集内，字符之间的比较关系。每一套字符集都有一套相应的校对规则/按照ASCLL码表大小比较规则（ci不区分大小写，cs区分大小写，bin二进制下区分，一般认为不区分），查看字符集：Show collation like ‘xxx’;
