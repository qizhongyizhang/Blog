MySQL进阶-查询优化

发表于 2021-05-18 更新于 2021-11-09 分类于 进阶 阅读次数： Valine：

写在前面：本文主要记录MySQL的查询优化策略

查询效率

影响因素

  • 数据量的多少
  • 硬件设备的性能
  • 磁盘IO数

外部表现

  • 大量请求阻塞
  • SQL操作变慢
  • 存储出现问题

优化策略

SQL层面-SQL优化

开启慢SQL记录

调优工具-生成执行计划：在查询语句之前加上 explain 关键字
SQL细节：尽量避免全表扫描

  • where 子句中，表连接写在最前面，可以过滤掉大量数据的条件写在最右边，条件匹配从右向左依次进行
  • 使用 exists 替代 in；[not] in -> [not] exists
  • 尽量避免在 where 子句中进行 null判断 和表达式操作，会导致放弃使用索引进行全表扫描
  • Distinct 替换为 group by，并结合 order by 使用
  • 通配符（like）和正则表达式匹配会尝试匹配表中所有行，应尽量避免使用
  • 避免在索引列上进行 null判断 和计算
  • 使用 join 代替 子查询

表结构层面-索引

索引 ：是帮助数据库高效获取数据的数据结构，是数据库优化中最常用也是最重要的手段之一，数据量越大，索引查询效率提高越明显

  • 使用条件

对于创建的多列索引，只要查询条件使用了最左边的列，索引一般就会被使用
对于使用like的查询，后面如果是常量并且只有%不在第一个字符，索引才可能被使用
如果使用col_name is null将使用索引
尽量采用自增主键作为主索引：自增主键是连续的，具有较少的页分裂，并且能减少数据移动(减少分裂和减少数据移动)

  • 禁用条件

如果条件中有or，即使其中有条件有索引也不会使用
对于多列索引，不是使用的第一部分，则不会使用索引
like查询是以%开头
如果列类型是字符串，那一定要在条件中将数据使用引号引用起来；否则不使用索引
如果mysql估计使用全表扫描要比使用索引快，则不使用索引

表结构层面-数据类型

字段类型优先级：整形 -> date、time - > enum、char -> varchar -> blob、text；优先考虑数字类型，其次是日期或者二进制类型，最后是字符串，同级别数据类型优先选择占用口昂见效的数据类型

数据库架构-分库分表

使用场景：单表数据超过500万或者表容量超过2G，推荐进行分库分表，如果预计三年后的数据量达不到级别，不要在创建时就分库分表；有垂直切分（按照业务切分，表结构可能不同）和水平切分（表结构相同，按照主键ID、时间等进行分表）两种
